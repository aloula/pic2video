# Feature Specification: Multiplatform Desktop GUI

**Feature Branch**: `010-create-feature-branch`  
**Created**: 2026-04-11  
**Status**: Draft  
**Input**: User description: "Create a basic GUI that can run multiplatform (Windows, Linux and MacOS). The interface must be clean and allow the user to select the input and output folder, keeping the default the current path where the application was launched. The user must also be able to select any application option (exif-overlay, kenburns-effect...). The GUI also needs to show a basic status information, like loading files, processing, finished. A small log box to show the program output is also welcome."

## Clarifications

### Session 2026-04-11

- Q: Should GUI output selection control full output filename/path or only destination folder? → A: Output folder only; filename remains auto-generated by profile.
- Q: Which option set should the GUI expose? → A: Expose all current user-facing render options.
- Q: How should GUI handle an input folder with no supported media files? → A: Block Start during preflight and show actionable validation error.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Configure Render Job Visually (Priority: P1)

As a desktop user, I can open a simple GUI, choose input and output folders, adjust rendering options, and start a run without memorizing CLI flags.

**Why this priority**: This is the core value of the feature. Without visual job configuration, the GUI does not solve the primary usability problem.

**Independent Test**: Can be fully tested by launching the GUI, selecting folders/options, starting a render, and confirming the resulting run uses exactly the selected values.

**Required Test Coverage**:

- Unit Tests: Default path initialization from launch directory; option-to-run configuration mapping; validation of empty or inaccessible paths.
- E2E Test: User opens GUI, selects folders and options, starts render, and output media is created in selected destination.

**Acceptance Scenarios**:

1. **Given** the app is launched from a folder, **When** the GUI opens, **Then** both input and output folder fields default to that launch folder.
2. **Given** the user changes input/output folders and toggles options (including EXIF overlay and image effect), **When** they start rendering, **Then** the run starts using those exact selections and writes output into the selected destination folder with the profile-based auto filename.
3. **Given** the GUI is open, **When** the user inspects render settings, **Then** all currently user-facing render options are available for selection.

---

### User Story 2 - Track Run Progress and Outcome (Priority: P2)

As a user running a render, I can see clear status states so I know whether the app is loading files, processing, finished, or failed.

**Why this priority**: Users need confidence and visibility during processing; status feedback reduces confusion and repeated retries.

**Independent Test**: Can be fully tested by starting a render and observing state transitions throughout the run until completion or failure.

**Required Test Coverage**:

- Unit Tests: Status state transitions and state-reset behavior between runs.
- E2E Test: User starts a render and sees status update from idle to loading to processing to finished (or failed if an error is induced).

**Acceptance Scenarios**:

1. **Given** the app is idle, **When** the user starts rendering, **Then** status updates to loading files before processing begins.
2. **Given** processing completes successfully, **When** work ends, **Then** status updates to finished and remains visible until a new run starts.

---

### User Story 3 - View Execution Log in GUI (Priority: P3)

As a user, I can see a compact log box with runtime output so I can understand progress details and diagnose failures.

**Why this priority**: Logging is secondary to job execution, but it significantly improves supportability and user trust.

**Independent Test**: Can be fully tested by running the app and verifying new process output lines appear in the log area in order.

**Required Test Coverage**:

- Unit Tests: Log line buffering/appending and maximum size behavior.
- E2E Test: User starts a run and sees startup, progress, and completion/error lines appear in the GUI log box.

**Acceptance Scenarios**:

1. **Given** a render has started, **When** the underlying process emits output, **Then** the log box displays new lines in chronological order.
2. **Given** a render fails, **When** an error line is emitted, **Then** the error message appears in the log box and status changes to failed.

### Edge Cases

- When the selected input folder contains no supported media files, Start is blocked and the GUI shows an actionable validation message.
- How does the system handle a selected output folder that cannot be written?
- What happens if the user starts a new run while a previous run is still processing?
- How does the system behave when the application is launched from a directory path that no longer exists?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide a desktop GUI that can be launched and used on Windows, Linux, and macOS.
- **FR-002**: System MUST show input-folder and output-folder selectors in the main interface.
- **FR-003**: System MUST initialize both input and output folder values to the current working directory from which the application is launched.
- **FR-004**: Users MUST be able to change input and output folders before starting a run.
- **FR-005**: System MUST expose selectable render options currently available to users (including EXIF overlay and image-effect selection).
- **FR-005a**: System MUST expose all current user-facing render options from the `render` workflow, not only a reduced subset.
- **FR-006**: System MUST preserve selected option values for the active run and apply them exactly when rendering starts.
- **FR-006a**: System MUST treat GUI output selection as destination folder selection only; output filename MUST remain auto-generated by selected profile.
- **FR-007**: System MUST display a visible run-status indicator with, at minimum, idle, loading files, processing, finished, and failed states.
- **FR-008**: System MUST update status in near real time as run state changes.
- **FR-009**: System MUST provide a log box showing runtime output lines generated during execution.
- **FR-010**: System MUST append new log lines in chronological order and keep the newest lines visible.
- **FR-011**: System MUST show user-actionable validation feedback before starting a run when required fields or paths are invalid.
- **FR-011a**: System MUST perform preflight input-folder checks and block run start when no supported media files are found.
- **FR-012**: System MUST prevent concurrent render starts from the same GUI window while one run is active.
- **FR-013**: System MUST define and maintain unit tests for all changed behaviors.
- **FR-014**: System MUST define and maintain end-to-end tests for primary user flows.

### Key Entities *(include if feature involves data)*

- **GuiRunConfiguration**: User-selected runtime configuration including input folder, output folder, selected options, and launch-time defaults.
- **GuiRunState**: Current execution state containing lifecycle status, timestamps, and final outcome.
- **GuiLogEntry**: A single ordered output line shown in the GUI log box, including text and capture time.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% of users can configure input/output folders and start a render from the GUI in under 60 seconds on first attempt.
- **SC-002**: 100% of completed runs show at least one status transition in each active phase (loading files, processing, completion state).
- **SC-003**: 100% of emitted runtime output lines are visible in the GUI log box in chronological order during active runs.
- **SC-004**: At least 90% of pilot users report that the GUI makes routine render setup easier than equivalent command-line usage.

## Assumptions

- The initial GUI scope is a single-window desktop interface focused on local rendering workflows.
- Existing render capabilities and options remain the source of truth; the GUI orchestrates them rather than redefining render behavior.
- Defaulting input/output to launch directory means using the process working directory at startup time.
- Localization and theming are out of scope for this first GUI version.
